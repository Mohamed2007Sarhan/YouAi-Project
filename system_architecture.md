# YouAI Digital Twin - Comprehensive System Documentation

This document provides a complete, step-by-step explanation of the YouAI Digital Twin project, including the system workflow, directory structure, and the specific role of each file in the project.

## 1. Project Overview

**YouAI** is an advanced "Digital Twin" system designed to create an autonomous AI representation of a user. It captures the user's personality, memories, writing style, and social media footprint, storing them in an encrypted local database. Using advanced Language Learning Models (LLMs), text-to-speech (TTS), speech-recognition, and system automation tools, the AI can interact, speak, and even control the desktop autonomously based on the user's identity.

## 2. Complete System Workflow (Step-by-Step)

The typical execution path of the application is as follows:

### Phase 1: Installation & Preparation
1. **Running `installer.py`:** Before anything else, the user runs the installer. It checks for Python version compatibility, verifies `pip`, creates necessary directories (e.g., `logs`, `Backend/data`, `Backend/memory`), generates an encrypted environment file (`.env.memory`), and installs required dependencies from `requirements.txt`.

### Phase 2: System Boot & Integrity Check
2. **Launching `Start.py`:** The main entry point of the program.
3. **Logging Initialization:** It sets up an advanced logging system that auto-cleans old logs and generates unique session logs.
4. **Memory Verification:** `Start.py` triggers `check_missing_info()` from the memory manager. It scans the SQLite database (`ai_giant_memory.db`) to see if the user's profile is complete (25 distinct personality/memory categories).

### Phase 3: Setup Wizard (If Data is Missing)
5. **Social Media Setup:** If the memory is entirely empty, the `SocialInputWindow` (GUI) is shown. The user inputs their social media links.
6. **Data Scraping (`SocialFetch`):** Background threads fetch public data from provided links (GitHub, Twitter, Reddit, etc.) using automated headless browsers and APIs.
7. **Profiling:** The LLM reads the scraped data and constructs an initial personality profile, storing it in the database.
8. **Deep Interview:** The AI analyzes which memory categories are still empty (e.g., "Childhood memories", "Fears", "Decision styles"). It generates targeted, personalized questions using the LLM.
9. **Voice/Text Q&A:** The `VoiceSetupWindow` prompts the user with these questions. The user's voice answers are transcribed (Speech-to-Text), profiled by the LLM, and injected into the memory database.

### Phase 4: Main System Execution
10. **The Twin Awakens:** Once the database has sufficient data, `Start.py` transitions to the `MainWindow`.
11. **Autonomous Operation:** The AI now acts as the user's twin. It listens for commands, utilizes long-term memory to maintain context, controls the UI/desktop through automation tools (`TaskPlanner`, `pyautogui`), and speaks back using TTS engines.

## 3. Directory Structure and File Roles

### 📂 Root Directory

*   **`Start.py`**
    *   **Role:** The core orchestrator and main entry point. 
    *   **Function:** Handles the logic of checking the database completion, deciding whether to show the setup wizard or the main chat window, routing data between the UI and the Backend, handling application state, and triggering background scraping/profiling threads.
*   **`installer.py`**
    *   **Role:** Environment preparation script.
    *   **Function:** Checks system requirements, creates folders, virtual environments, and installs required packages safely without altering source code.
*   **`README.md`**
    *   **Role:** The standard quick-start guide for the project.
*   **`requirements.txt`**
    *   **Role:** Lists all necessary Python dependencies (e.g., `PyQt6`, `pyautogui`, `opencv-python`).
*   **`add_age.py`, `db_utils.py`, `view_memory.py`**
    *   **Role:** Utility scripts.
    *   **Function:** Independent scripts usually used by the developer/admin to directly view or inject specific data into the SQLite database for debugging or manual overrides.

### 📂 `FrontEnd/` (User Interface & Media)

*   **`gui/setup_windows.py`**
    *   **Role:** First-time user experience interfaces.
    *   **Function:** Contains PyQt6 classes like `SocialInputWindow` (for social links), `VoiceSetupWindow` (for the deep interview), and `LoadingWindow`.
*   **`gui/main_window.py`**
    *   **Role:** The primary application interface.
    *   **Function:** The chat and visualization interface used when the twin is fully functional and ready to interact.
*   **`audio/speech_engine.py`**
    *   **Role:** Speech-to-Text (STT).
    *   **Function:** Listens to the user's microphone and converts spoken words into text strings.
*   **`audio/tts_engine.py`**
    *   **Role:** Text-to-Speech (TTS).
    *   **Function:** Takes text generated by the LLM and speaks it aloud through the system's speakers.

### 📂 `Backend/` (Core Logic & AI)

*   **`services/twin_orchestrator.py`**
    *   **Role:** The brain's manager.
    *   **Function:** Receives raw data (like social media dumps or user answers), passes it to the LLM to extract personality traits, and maps those traits to the correct database schemas.
*   **`services/auth_connector.py` & `reset_manager.py`**
    *   **Role:** Security and maintenance.
    *   **Function:** Manages system security states and provides safe ways to reset the digital twin's memory if the user wants to start over.
*   **`memory/ai_giant_memory.db`**
    *   **Role:** The brain's storage.
    *   **Function:** A highly structured local database containing 25+ tables (e.g., cognitive_profile, emotional_patterns, decision_history).
*   **`memory/memory_management.py` & `memory_schema.py`**
    *   **Role:** Database operators.
    *   **Function:** Defines the strict table structures (schemas) and provides functions to safely Read, Write, Update, and encrypt memory blocks.
*   **`memory/persona_builder.py`**
    *   **Role:** Identity synthesis.
    *   **Function:** Pulls scattered data from the database and compiles it into a unified "system prompt" or context block so the LLM knows *who* it is supposed to be.
*   **`memory/question_generator.py`**
    *   **Role:** Setup assistant.
    *   **Function:** Identifies missing data in the database and generates appropriate prompts to ask the user.
*   **`SocialFetch/`** (Folder)
    *   **Role:** OSINT (Open Source Intelligence) scrapers.
    *   **Function:** Contains separate scripts (`github.py`, `twitter.py`, `reddit.py`, etc.) designed to visit social media profiles, bypass simple protections, and extract public bios, posts, and interaction styles to feed the memory engine.
*   **`tools/task_planner.py` & `tools/smart_fetch.py`**
    *   **Role:** Autonomous Actions.
    *   **Function:** Grants the AI agency. `task_planner` allows the AI to break down complex requests into steps and execute physical OS actions (clicking, typing). `smart_fetch` uses browser cookies to pull data more securely.
*   **`core/autonomous_agent.py` & `core/mood_manager.py`**
    *   **Role:** Behavioral regulation.
    *   **Function:** `mood_manager` tracks the AI's emotional state based on recent interactions (e.g., happy, annoyed) which alters its responses. `autonomous_agent` dictates when the AI should act on its own versus waiting for a prompt.
*   **`llms/nvidia_llm.py`** (and others)
    *   **Role:** Model connection.
    *   **Function:** Handles the API calls to the actual neural network models (like NVIDIA, Groq, etc.), sending the context and receiving the generated text.

## 4. How the AI Learns (Data Flow)

1. **Input:** User talks to `Start.py` OR `SocialFetch` scrapes a website.
2. **Analysis:** The raw text is sent via `twin_orchestrator.py` to `nvidia_llm.py`.
3. **Extraction:** The LLM is instructed: *"Read this text. Extract the user's political views, humor style, and fears in JSON format."*
4. **Storage:** `memory_management.py` takes the JSON, encrypts it, and saves it into specific tables in `ai_giant_memory.db`.
5. **Recall:** Later, when the user asks a question, `persona_builder.py` fetches these records, bundles them up, and hands them back to the LLM so the AI answers *exactly* as the user would.
